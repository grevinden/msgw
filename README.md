# 🎉 MessageCenter – твой неубиваемый почтальон

Представь: ты отправляешь важное сообщение, а друг отключился. Обидно?  
С **MessageCenter** не страшно — мы **сохраним его в Redis**, и как только друг появится онлайн, он **всё получит**. А если не появится — сообщение само умрёт через заданное время (TTL). Магия? Нет, просто крутой шлюз.

---

## 🚀 Запуск за 30 секунд

Всё, что нужно — **Docker** и пара команд.

```bash
# Скачай образ
docker pull ghcr.io/grevinden/msgw:latest

# Запусти вместе с Redis (они в одной сети)
docker network create msgw-net
docker run -d --name redis --network msgw-net redis --appendonly yes
docker run -d --name msgw --network msgw-net -p 8000:8000 \
  -e MSGW_CACHE=redis://redis:6379/0 \
  ghcr.io/grevinden/msgw:latest
```

Готово! Твой шлюз слушает `ws://localhost:8000/любой_путь` и `http://localhost:8000/любой_путь`.

## 📬 Как общаться через WebSocket

Протокол — обычный JSON. У каждого сообщения есть **uuid** (уникальный ключ), **ttl** (время жизни) и **payload** — содержимое.

⚠️ **Важная механика**: в Redis под каждым `uuid` может храниться **только одно сообщение**. Если ты отправил `send`, а потом кто‑то отправил `done` с тем же `uuid` – `send` **затрётся**. То же самое с `fail`. Последнее присланное сообщение по ключу всегда побеждает.

### Типы пакетов

- **`send` – «Привет всем!» 👋**  
  Сохраняет этот JSON под ключом `uuid` на `ttl` (или на `MSGW_CACHE_TTL`, если не указан) → рассылает всем.

- **`done` – «Всё, принял, спасибо!» ✅**  
  **Перезаписывает** тот же ключ `uuid` своим содержимым на `MSGW_CACHE_TTL` секунд → рассылает всем.

- **`fail` – «Ой, что‑то пошло не так…» 😵‍💫**  
  **Перезаписывает** тот же ключ `uuid` своим содержимым на `MSGW_CACHE_TTL` секунд → рассылает всем.

> 🧠 Таким образом, `done` и `fail` уничтожают исходный `send`. Новый клиент, подключившись, увидит только **последнее** сообщение по каждому `uuid`. Квитанции живут столько же, сколько и обычные сообщения (по умолчанию 3600 секунд).

## 📦 Типы пакетов (с примерами)

Все пакеты — JSON. У каждого есть обязательные поля:

- **`uuid`** — уникальный идентификатор (строка, например `"123e4567-e89b-12d3-a456-426614174000"`).
- **`ttl`** — время жизни в секундах (целое положительное число). Если не указано, используется значение из переменной окружения `MSGW_CACHE_TTL`.
- **`payload`** — объект с полем `typ` и дополнительными данными.

> 🧠 **Важно**: под одним `uuid` в хранилище может быть только одно сообщение. `done` и `fail` затирают `send` с тем же `uuid`.

---

### 1️⃣ `send` – обычное сообщение

```json
{
  "uuid": "123e4567-e89b-12d3-a456-426614174000",
  "ttl": 3600,
  "payload": {
    "typ": "send",
    "top": "новости",
    "mes": "Сервер перезапущен!"
  }
}
```

- **Что значит**: «Эй, всем привет!» — требует подтверждения.
- **Что делает сервер**: сохраняет этот JSON в Redis под ключом `uuid`, рассылает всем подключённым.
- **TTL**: если не указан, берётся `MSGW_CACHE_TTL`. По истечении TTL сообщение автоматически удаляется.

---

### 2️⃣ `done` – квитанция об успешной обработке

```json
{
  "uuid": "123e4567-e89b-12d3-a456-426614174000",
  "ttl": 3600,
  "payload": {
    "typ": "done"
  }
}
```

- **Что значит**: «Всё, принял, спасибо!» — сообщение обработано.
- **Что делает сервер**: **перезаписывает** ключ `uuid` этим JSON (живёт `ttl` секунд) и рассылает всем.
- **Результат**: исходный `send` уничтожается, новый клиент увидит `done`.

---

### 3️⃣ `fail` – квитанция об ошибке

```json
{
  "uuid": "123e4567-e89b-12d3-a456-426614174000",
  "ttl": 3600,
  "payload": {
    "typ": "fail",
    "err": "Не удалось разобрать JSON"
  }
}
```

- **Что значит**: «Ой, что‑то пошло не так…».
- **Когда отправляется**: либо автоматически сервером при невалидном JSON, либо клиентом при своей внутренней ошибке.
- **Что делает сервер**: **перезаписывает** ключ `uuid` этим JSON (живёт `ttl` секунд) и рассылает всем.
- **Результат**: исходный `send` уничтожается, все узнают об ошибке.

> ⚡ **Важно про `ttl`**: если в сообщении не указать `ttl`, сервер использует значение `MSGW_CACHE_TTL` (по умолчанию 3600 секунд). Это работает и для `send`, и для `done`, и для `fail`.

А код на JS такой:

```javascript
const ws = new WebSocket("ws://localhost:8000/любой_путь");
ws.onopen = () => ws.send(JSON.stringify({
  uuid: crypto.randomUUID(),
  ttl: 3600,
  payload: { typ: "send", top: "прикол", mes: "лооол" }
}));
ws.onmessage = (e) => {
  const m = JSON.parse(e.data);
  if (m.payload.typ === "send") {
    console.log(`📨 ${m.payload.top}: ${m.payload.mes}`);
    // подтверждаем – затираем исходное сообщение
    ws.send(JSON.stringify({
      uuid: m.uuid,
      ttl: m.ttl,
      payload: { typ: "done" }
    }));
  }
};
```

> ⚡ При новом подключении сервер **отдаёт текущее содержимое** каждого ключа. Если под ключом лежит `send` – отдаст `send`, если его перезаписали `done` – отдаст `done`. Никакой магии.

## 🎛️ Настройка через переменные окружения

Ты можешь менять поведение шлюза, не пересобирая образ. Всё через `-e` в `docker run`.

### Основные (твои любимые)

- **`MSGW_CACHE`**  
  По умолчанию: `mem://?check_interval=1`  
  Где хранить сообщения. `redis://redis:6379/0` — надёжно и насовсем.

- **`MSGW_CACHE_BATCH_SIZE`**  
  По умолчанию: `100`  
  Сколько ключей обрабатывать за раз при сканировании (не трогай, если не уверен).

- **`MSGW_CACHE_TTL`**  
  По умолчанию: `3600`  
  Время жизни для **всех** сообщений (и send, и receipt), если в самом сообщении нет `ttl`.

- **`APP`**  
  По умолчанию: `MSGW`  
  Меняет префикс всех переменных (хитро для нескольких экземпляров).

### Докерные плюшки (внутренности)

Эти переменные уже прописаны в образе, но ты можешь переопределить их при запуске, если знаешь, что делаешь.

- **`PYTHONOPTIMIZE=1`** – выкидывает `assert` — быстрее и легче.  
- **`TZ=Europe/Moscow`** – часовой пояс для логов.  
- **`UV_NO_DEV=1`** – не ставит dev-зависимости (экономит место).  
- **`UV_FROZEN=1`** – строго использует `uv.lock` — без сюрпризов.  
- **`UV_LINK_MODE=copy`** – копирует файлы вместо жёстких ссылок (нужно для Docker в некоторых окружениях).

### Настройки Uvicorn (серверная часть)

Хочешь поменять порт, количество воркеров или частоту WebSocket‑пингов? Вот тебе переменные.

- **`UVICORN_HOST`** (по умолч. `0.0.0.0`) – слушаем все интерфейсы (нужно для Docker).  
- **`UVICORN_PORT`** (по умолч. `8000`) – порт.  
- **`UVICORN_WORKERS`** (по умолч. `1`) – воркеров. Для асинхронного приложения лучше не трогать.  
- **`UVICORN_LOG_LEVEL`** (по умолч. `info`) – логи: `debug`, `info`, `warning`, `error`.  
- **`UVICORN_ACCESS_LOG`** (по умолч. `true`) – логировать каждый запрос.  
- **`UVICORN_WS`** (по умолч. `websockets-sansio`) – протокол WebSocket (не лезь, если не знаешь).  
- **`UVICORN_WS_PING_INTERVAL`** (по умолч. `3`) – как часто посылать пинги клиентам (сек). Увеличь, если хочешь меньше трафика.  
- **`UVICORN_WS_PING_TIMEOUT`** (по умолч. `2`) – таймаут ожидания понга.  
- **`UVICORN_WS_PER_MESSAGE_DEFLATE`** (по умолч. `1`) – сжимать сообщения (экономит трафик).  
- **`UVICORN_TIMEOUT_KEEP_ALIVE`** (по умолч. `3`) – keep‑alive для HTTP.  
- **`UVICORN_TIMEOUT_GRACEFUL_SHUTDOWN`** (по умолч. `60`) – сколько ждать завершения запросов при остановке.

Пример с кастомными параметрами:

```bash
docker run -d --name msgw -p 8080:8080 \
  -e MSGW_CACHE=redis://redis:6379/0 \
  -e MSGW_CACHE_TTL=7200 \
  -e UVICORN_PORT=8080 \
  -e UVICORN_WS_PING_INTERVAL=120 \
  ghcr.io/grevinden/msgw:latest
```

## 🧠 Что умеет MessageCenter (в реальной жизни)

- **Чат с историей для новичков** — зашёл, а тебе сразу прилетели все непрочитанные сообщения.
- **Очередь команд для ботов** — бот отвалился, команда не потерялась, переподключился — получил.
- **Логи в реальном времени** — разошли событие всем подписчикам.
- **Синхронизация микросервисов** — один кинул POST, другой тут же получил через WebSocket.

---

## 🔥 Почему это круто (не хвастовство, а факты)

- 🧠 **Redis на диске** — данные не сгорают при перезапуске.
- ⚡ **Асинхронность** — тысячи клиентов не тормозят друг друга.
- 🧩 **Валидация Pydantic** — кривые сообщения не пройдут, и ты узнаешь почему.
- 🧹 **Самозачистка** — мёртвые WebSocket‑соединения вылетают из пула автоматически.
- ⏲️ **TTL** — сообщения не висят вечно, сами умирают.
- 🐳 **Docker из коробки** — запустил и забыл.

## 🤙 Короче, ребята

1. Запустил контейнер → подключился по WebSocket → получил **текущее значение** по каждому ключу (то, что лежит в Redis под этим `uuid`).
2. Отправил `send` → сервер сохранил под ключом `uuid` на `ttl` (или `MSGW_CACHE_TTL`) и разослал всем.
3. Кто получил — может отправить `done` (или `fail`) с тем же `uuid`.  
   Сервер **перезаписывает** этим значением ключ `uuid` (снова на `MSGW_CACHE_TTL`) и рассылает его всем.
4. Исходный `send` исчезает (затёрт `done`). Никакого «долгого хранения» после подтверждения нет.

Всё просто, элегантно и надёжно.

**MessageCenter** — твой надёжный друг в мире сообщений. Пользуйся с удовольствием! 🫶
