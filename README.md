# 🎉 MessageCenter !

Твой личный **шлюз сообщений** с суперсилой: сообщения не теряются, даже если ты отключился! 🔥

## 💡 Что тут вообще происходит?

- Клиенты стучатся через **WebSocket** (живой канал туда‑сюда) или кидают сообщения через **HTTP POST** (для серверов).
- Сервер **запоминает** все важные сообщения (`notify`) в **Redis**. Перезагрузился? Не беда, данные останутся! 💾
- Когда приходит новичок, ему **сразу отправляют всю историю** неподтверждённых сообщений.
- А когда ты говоришь **«всё, получил»** (кидаешь `receipt`), сообщение удаляется из хранилища и больше не надоедает новым клиентам.
- Каждое сообщение **улетает всем**, кто сейчас в чате (broadcast). 🚀
- Каждое сообщение имеет **время жизни (TTL)** – автоматически удаляется, если не подтверждено.

## 🐳 Быстрый старт (Docker)

```bash
docker pull ghcr.io/grevinden/msgw:latest
docker run -d --name msgw -p 8000:8000 \
  -e MSGW_CACHE=redis://redis:6379/0 \
  ghcr.io/grevinden/msgw:latest
```

Для полноценной работы с постоянным хранилищем запустите Redis:

```bash
docker run -d --name redis -p 6379:6379 redis --appendonly yes
```

Затем свяжите контейнеры (например, через `--link` или общую сеть).

## 📦 Типы пакетов (сообщения)

Все пакеты – JSON. У каждого есть обязательные поля: `uuid`, `ttl` (время жизни в секундах) и `payload` – содержимое.

### 1️⃣ send – «Эй, всем привет!» 👋

```json
{
  "uuid": "123e4567-e89b-12d3-a456-426614174000",
  "ttl": 3600,
  "payload": {
    "typ": "send",
    "top": "новости",
    "mes": "Сервер перезапущен!"
  }
}
```

- **Что значит**: обычное сообщение, которое нужно доставить всем и получить подтверждение.
- **Что делает сервер**: сохраняет его в Redis (на время `ttl`), а затем рассылает всем подключённым.
- **Что надо сделать тебе**: когда получишь такое – обработай и пришли ответ `done` с тем же `uuid`.

### 2️⃣ done – «Всё, принял, спасибо!» ✅

```json
{
  "uuid": "123e4567-e89b-12d3-a456-426614174000",
  "payload": {
    "typ": "done"
  }
}
```

- **Что значит**: квитанция об успешной обработке.
- **Что делает сервер**: удаляет сообщение из Redis и рассылает `done` всем (чтобы все знали, что задача решена).

### 3️⃣ fail – «Ой, что‑то пошло не так…» 😵‍💫

```json
{
  "uuid": "123e4567-e89b-12d3-a456-426614174000",
  "payload": {
    "typ": "fail",
    "err": "Не удалось разобрать JSON"
  }
}
```

- **Генерируется автоматически**, если сервер не смог прочитать присланную JSON‑строку.
- **Клиент тоже может отправить `fail`** – сервер воспримет его как `receipt` (удалит `notify` из хранилища) и разошлёт всем.

> **Примечание о `ulid`:** ранее `ulid` вычислялся из `uuid`, теперь он генерируется как новый независимый идентификатор. Не используйте его для связи с `uuid` – поле оставлено для статистики и логов.

## 🔌 Как подключиться? (WebSocket)

```javascript
const ws = new WebSocket("ws://localhost:8000/любой_путь");

ws.onopen = () => {
  console.log("🔗 Влетели!");
  ws.send(JSON.stringify({
    uuid: crypto.randomUUID(),
    ttl: 3600,
    payload: { typ: "send", top: "привет", mes: "Кто тут?" }
  }));
};

ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  if (msg.payload.typ === "send") {
    console.log(`📨 ${msg.payload.top}: ${msg.payload.mes}`);
    ws.send(JSON.stringify({
      uuid: msg.uuid,
      ttl: msg.ttl,
      payload: { typ: "done" }
    }));
  }
};
```

> **Важно:** как только открыли соединение – сервер сразу высылает **все старые неподтверждённые сообщения** (только те, чей TTL не истёк).

## 🌍 Переменные окружения

Приложение настраивается через переменные с префиксом `MSGW_` (или `APP_`, если задан `APP`).

### Основные настройки MessageCenter

| Переменная | Тип | По умолчанию | Описание |
|------------|-----|--------------|-----------|
| `MSGW_CACHE` | URL | `mem://?check_interval=1` | URL хранилища кеша. Поддерживаются схемы `mem://`, `redis://`. Для Redis (рекомендуется) используйте `redis://redis:6379/0`. |
| `MSGW_CACHE_BATCH_SIZE` | положительное int | `100` | Количество ключей, обрабатываемых за раз при сканировании. |
| `MSGW_CACHE_TTL` | положительное int | `3600` | Время жизни ключей по умолчанию (секунды). Используется, если в сообщении не указано `ttl`. |
| `APP` | строка | `MSGW` | Изменяет префикс переменных (например, `APP=MYCENTER` → `MYCENTER_CACHE`). |

### Настройки Uvicorn (опционально)

Uvicorn также можно настраивать через переменные окружения с префиксом `UVICORN_`. Наиболее важные:

| Переменная | По умолчанию | Описание |
|------------|--------------|-----------|
| `UVICORN_HOST` | `0.0.0.0` | Адрес прослушивания. |
| `UVICORN_PORT` | `8000` | Порт. |
| `UVICORN_WORKERS` | `1` | Количество воркер‑процессов (для asyncio обычно 1). |
| `UVICORN_LOG_LEVEL` | `info` | Уровень логов (`debug`, `info`, `warning`, `error`). |
| `UVICORN_WS` | `websockets-sansio` | WebSocket протокол (`websockets-sansio`, `wsproto`, `auto`). |

Полный список переменных Uvicorn и их значений см. в [официальной документации](https://www.uvicorn.org/settings/).

Пример `docker run` с переменными:

```bash
docker run -d --name msgw -p 8000:8000 \
  -e MSGW_CACHE=redis://redis:6379/0 \
  ghcr.io/grevinden/msgw:latest
```

## 🎯 Какие задачи можно решать?

| Задача | 💡 Как это работает |
|--------|----------------------|
| **Чат с гарантией доставки** | Отправил `send` – все сразу видят. Когда прочитали – кидают `done`. Новый участник получает все непрочитанные сообщения. |
| **Уведомления для фронтенда** | Сервер шлёт `send` через POST, а все открытые веб‑морды получают всплывашки. |
| **Очередь команд для ботов/устройств** | Команда сохраняется до тех пор, пока устройство не подтвердит выполнение. Если устройство перезагрузилось – подключается и получает всё, что не сделано. |
| **Логи в реальном времени** | Сервер логирует события и отправляет их всем, кто подписан. |
| **Синхронизация между микросервисами** | Один сервис отправляет событие через POST, другие получают через WebSocket и реагируют мгновенно. |

## 🔥 Фишки

- 🧠 **Умное хранилище** – Redis с AOF (постоянное), данные не теряются.
- ⚡ **Асинхронность до мозга костей** – много клиентов не тормозят друг друга.
- 🧩 **Схема сообщений** – чёткая, простая, на Pydantic (валидация на уровне ядра).
- 🧹 **Самозачистка** – мёртвые WebSocket‑соединения вылетают из пула автоматически.
- 📡 **Любой путь** – не важно, какой URL для WebSocket или POST, всё работает.
- ⏲️ **TTL** – каждое сообщение автоматически удаляется, если не подтверждено за отведённое время.

## 🧙‍♂️ Коротко о главном

1. Подключаешься по WebSocket → получаешь все висящие `notify`.
2. Шлёшь `send` → сервер сохраняет и кидает всем.
3. Получил уведомление → отправь `done` (или `fail`) с тем же `uuid`.
4. Нет `done` → сообщение будет жить до истечения TTL и приходить каждому новому.
5. Можно слать сообщения и через обычный HTTP POST.

## 🎁 Бонус: пример жизни сообщения

1. **Клиент А** отправляет `send` на тему «котики» с `ttl=1800`.
2. Сервер сохраняет в Redis с TTL 30 минут и рассылает всем.
3. **Клиент Б** подключается через 5 минут – сервер отправляет ему то же сообщение (пока TTL не истёк).
4. **Клиент Б** обрабатывает и шлёт `done`.
5. Сервер удаляет запись о «котиках» – больше это сообщение никому не придёт.
6. Если бы `done` не пришёл, сообщение само бы удалилось через 30 минут.
